# Living Brain Research Notes

This file records research anchors for current architecture work. It is not a proof that MARULHO is a biological brain; it is the paper trail for the engineering direction.

## Current Anchors

### Predictive spiking substrate

- Persistent text-tick executor note, June 2026: profiling the promoted graph path showed that reconstruction scalar readback and Python neuromodulation still split each eligible text tick into two graph replays. NVIDIA's CUDA Graph programming guide states that whole-workflow submission reduces CPU launch cost and enables optimizations unavailable to piecewise stream submission; NVIDIA Holoscan's GPU-resident execution guidance similarly targets CPU/GPU coordination boundaries. MARULHO therefore moved reconstruction-driven dopamine, serotonin, acetylcholine, norepinephrine, and predicted-error updates onto persistent CUDA state and captured reconstruction, neuromodulation, fused route/vote, and transition as one replay. Three-factor SNN learning research supports neuromodulators as global factors over local plasticity, but does not justify graphing archival memory or replacing local spike state. Sixteen sequential ticks preserved every winner and reconstruction; all model tensors were exact except three velocity values with maximum `5.82e-11` drift. A fresh-process 512-tick comparison reached `95.58` versus `63.49 ticks/sec` (`1.506x`), while the text quality gate reached `57.14` versus `47.52 ticks/sec` (`1.202x`) with exact winners and preserved declared quality. One real 24-token source tick reported 24 persistent replays, 24 host truth synchronizations, and zero failures, but only `8.79 tokens/sec`; memory, cross-modal/text, source, and other Python stages remain the next target. Sources: [CUDA Graphs](https://docs.nvidia.com/cuda/cuda-programming-guide/04-special-topics/cuda-graphs.html), [GPU Resident Execution](https://docs.nvidia.com/holoscan/sdk-user-guide/using-the-sdk/gpu-resident-execution), and [Three-Factor Learning in Spiking Neural Networks](https://arxiv.org/abs/2504.05341).
- Fused reconstruction note, June 2026: CUDA Graph and Triton guidance still points toward removing duplicated fixed-shape GPU work rather than adding host-visible reports. MARULHO found that the persistent text tick computed reconstruction with a separate dense prototype scan even though fused route/vote had already scored the same exact routing cache. The promoted path now derives reconstruction error from the fused route-score maximum before device neuromodulation, preserving exact winners and bounded `1e-6` reconstruction tolerance. A pinned-clean-HEAD 1024-sample comparison measured `796.22` versus `630.17 tokens/sec` on the quantum-staged complete encoded tick (`1.264x`), with `1088` graph replays, `1088` fused reconstruction updates, zero graph failures, and zero staged-input mismatches. This changes implementation direction only inside the graph-owned text-tick boundary; sensory ticks, source handling, archival memory, replay, and service orchestration remain outside the claim.

- Thousand Brains Project / Monty note, June 2026: Monty's learning-module direction treats cortical-column-like modules as repeatable sensorimotor modelers that communicate through a common messaging protocol and reference-frame evidence. MARULHO borrows the bounded module/vote/metabolism shape without importing Monty as a runtime dependency. `core.column_runtime` keeps awake budgets, cached votes, disagreement, growth, and pruning evidence report-only. The report scheduler still does not control execution or topology.
- Column-runtime implementation note, June 2026: Thousand Brains / Monty sources reinforced the repeated learning-module and voting contract, while Switch Transformer-style sparse routing reinforced keeping active modules few and capacity-bound. MARULHO therefore adds sampled Column Registry evidence and scheduler/vote metadata without promoting scheduler execution. - Predictive-vote metabolism note, June 2026: Monty's voting contract and sparse-MoE literature support limiting active specialists, but they do not imply that every mathematically sparse operation is faster on a GPU. ## Engineering Implications

- Keep **Subcortex** tensor state on the configured `torch.device`.
- Preserve sparse/event structure where possible before reaching for dense CUDA kernels.
- Record actual device/backend evidence in reports before claiming CUDA acceleration.
- For routing, separate configured intent (`search_device`) from observed cache placement (`torch_vector_cache_device`) and benchmarked execution. Removed experimental backends should stay out of the live config surface until fresh evidence justifies a new implementation.
- For predictive/plasticity/binding work, local traces and eligibility state are the auditable CUDA surface. Device reports should prove where those live tensors reside before performance claims.
- For neuron dynamics, report membrane voltage, adaptation, and spike-time tensors directly, and preserve them across checkpoints without silently restoring to CPU.
- For cross-modal grounding, report the live devices of text/visual/audio traces, cross-modal weights, and confidence state.
- For spatial or hypercube binding, report topology tensor devices as well as learned binding-state devices; sparse topology claims need observable placement.
- For context and abstraction, report recurrent/context tensors, adaptive timescales, slow abstraction state, and top-down feedback tensors because they directly gate routing and curiosity.
- For text encoders, report learned chunking codebooks, semantic bucket embeddings, adapters, feature vectors, and spike traces on the selected device while keeping string parsing and segmentation as control-plane work.
- Keep memory consolidation storage CPU-resident unless the operation is active replay computation. Archival buffers, tags, timestamps, and raw windows are evidence records; replay tensors should move to the model device when consumed.
- Carry sensory encoder device metadata and last emitted spike tensor device/shape into grounded observations and previews so CUDA sensory claims are auditable.
- Move generated and directory-loaded visual/audio tensors to the resolved runtime device before sensory encoding; `map_location="cpu"` is archival behavior, not a CUDA-first sensory path.
- Active checkpoint restore should resolve `torch.load` to the runtime device. CPU archival saves remain acceptable for portability, but a live restored trainer must re-enter the Subcortex path through the selected device, not through a compatibility CPU default.
- Surface trainer-owned encoder device reports in the operator-facing runtime evidence, because model-only CUDA scope is incomplete when the active input encoder is owned by the trainer/service layer.
- Treat **Living Brain** as an evidence-gated runtime target: tick, train, think, replay, act, sleep, and runtime truth must be observable.
- Treat the former **Cortex Backend** as deleted from active runtime claims. LLM/NIM adapters must not remain as compatibility or experiment code while Subcortex/world-model paths become the real cognition surface.
- Make weak paths retireable: latency, external dependency, grounding quality, and liveness contribution should be visible before a path earns permanence.

- Bounded replay-window note, June 2026: modern Hopfield networks show attention-like associative retrieval can be a powerful local memory operator, but MARULHO treats that as bounded column/replay-window recall rather than a transformer-style global mind. Complementary learning systems and continual-learning replay support separating fast online learning from slower replay/consolidation; synaptic tagging/capture supports replay only when tag/PRP evidence marks a trace as worth stabilizing; latent and sparse replay work supports compact, selected rehearsal rather than GPU-resident archival metadata.
